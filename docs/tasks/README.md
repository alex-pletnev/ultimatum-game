# Task-трекер

Простой файловый трекер задач. Одна задача = один MD-файл. Индекс — [INDEX.md](INDEX.md).

## Формат имени файла

```
T-XXX-kebab-case-title.md
```

- `XXX` — номер задачи (000-padded, `T-001`, `T-042`).
- Заголовок — kebab-case, латиница, коротко.

Пример: `T-001-apply-index-sql-on-startup.md`.

## Структура таск-файла

```markdown
---
id: T-XXX
title: Полное название задачи
status: pending | in_progress | blocked | done | cancelled
priority: low | medium | high
created: YYYY-MM-DD
updated: YYYY-MM-DD
related_code:
  - src/main/kotlin/edu/itmo/ultimatum_game/services/SessionService.kt
related_docs:
  - docs/04-services.md
tags: [backend, api, security, db, refactor, bug, feature, tech-debt, meta]
---

## Контекст

Зачем это нужно. Ссылка на источник (`docs/11-known-gaps.md`, инцидент, обсуждение).

## Acceptance criteria

Что считать сделанным. Пункты чек-листа:

- [ ] Условие 1
- [ ] Условие 2

## План

Пошагово. Ссылки `file:line`.

1. Шаг 1
2. Шаг 2

## Лог

- `YYYY-MM-DD`: событие / решение / блокер.
```

## Статусы

| Статус | Когда |
|--------|-------|
| `pending` | Задача заведена, работа не начата |
| `in_progress` | Кто-то работает над задачей |
| `blocked` | Нужно решение или внешнее событие; причина в логе |
| `done` | Acceptance criteria выполнены, `related_docs` обновлены |
| `cancelled` | Отменена; причина в логе |

## Правила обновления

- `updated` меняется **при каждой правке** таск-файла.
- Любое существенное изменение фиксируется в `## Лог`.
- Перевод в `done` — только через `/task-done` (или ручной чек: acceptance выполнены, docs синхронизированы, INDEX обновлён).
- Отмена — `status: cancelled`, причина в логе. Файл не удалять.

## Slash-команды

| Команда | Что делает | Git |
|---------|-----------|-----|
| `/task-add "название"` | Создаёт `T-XXX-*.md`, обновляет INDEX.md | commit (без push) |
| `/task-done T-XXX` | Меняет `status: done`, лог, INDEX.md | тесты → commit → push |
| `/task-sync` | Проверяет консистентность INDEX ↔ файлов ↔ кода | нет (только чтение) |
| `/docs-sync` | Актуализирует `docs/*.md` по текущему `src/` | commit → push (если правки применены) |

Полные правила git-flow — `CLAUDE.md` → раздел «Git-автоматизация».

## Специальные теги

| Тег | Смысл |
|-----|-------|
| `tech-debt` | Проблема кода, замеченная агентом по ходу другой работы. Не блокирует текущую задачу, но должна быть починена. |
| `meta` | Улучшение самого инструментария: `CLAUDE.md`, `.claude/skills/`, формат таск-файла, структура `docs/`. |

## Auto-mode: заведение задач агентом

Агент имеет право сам заводить задачи, если по ходу другой работы заметил проблему или идею — но с **фильтром из трёх условий** (все должны быть верны):

1. Проблема реальная (сломанное поведение, скрытый риск, дублирование, нарушение инварианта) — не косметика.
2. Формулируется `## Контекст` в 2-3 строки и минимум один пункт `## Acceptance criteria`.
3. Проблема не относится к текущей задаче и растворится без записи.

Если хоть одно не выполнено — задача **не** заводится, вместо этого агент упоминает наблюдение одной строкой в ответе.

Обязательные атрибуты авто-заведённой задачи:
- Тег `tech-debt` или `meta`.
- Приоритет `low` по умолчанию, `medium` для явно опасных случаев, `high` — только по согласованию с пользователем.
- В `## Лог` — первая запись: `- YYYY-MM-DD: заведена автоматически в ходе работы над T-XXX (или без родительской задачи, если работа была ad-hoc).`

Правило прозрачности: агент отдельной строкой в конце своего ответа сообщает: «Попутно завёл T-XXX (…): краткое описание. Отменить — скажи». Пользователь может отменить одной фразой.

Полные правила — в `CLAUDE.md` → раздел «Проактивное заведение задач».

## Заметки для агента

- Задача рождается из явного запроса пользователя, из пункта `docs/11-known-gaps.md` или из Auto-mode.
- Перед созданием — проверить, нет ли похожей открытой задачи (`grep` по INDEX и файлам).
- `related_code` / `related_docs` — обязательные поля: без них `/docs-sync` и `/task-sync` не смогут корректно работать.
