## Контекст

В T-017 агент запустил `./gradlew check`, тест `UltimatumGameApplicationTests` завис на ~20 минут (Docker daemon был выключен, `spring-boot-docker-compose` бесконечно ждал Postgres). Всё это время агент послушно ждал завершения, ставил wakeup'ы каждые 90 секунд, объяснял «первый запуск с новыми зависимостями идёт долго» — то есть трактовал аномалию как норму.

Правильное поведение — заподозрить неладное гораздо раньше: у этого проекта unit-тесты обычно проходят за <60 секунд, а Spring context load — за <30 секунд. Всё что дольше 3–5 минут при обычной работе — уже аномалия, а не «просто долго компилится».

## Acceptance criteria

- [ ] В `CLAUDE.md` появляется раздел (или правило в существующем) про эвристики ожидания долгих команд: типичные baselines для этого проекта + порог «пора эскалировать пользователю».
- [ ] Правило конкретное: указаны команды (`./gradlew check`, `./gradlew test`, `./gradlew bootRun`), их типичное время, порог тревоги.
- [ ] Правило указывает первую диагностическую реакцию: посмотреть stdout/stderr процесса, проверить внешние зависимости (Docker daemon, порты БД), рассказать пользователю что странно.

## Возможные формулировки правила

```
## Долгие команды — эвристика ожидания

Baselines для этого проекта:
- ./gradlew compileKotlin — ~10s
- ./gradlew test (unit) — ~30s
- ./gradlew check (с прогретым кэшем) — ~60s, первый запуск после новых зависимостей — до 3 min
- ./gradlew bootRun (до RUNNING) — ~20s
- @SpringBootTest контекст-лоад — ~15s

Если команда идёт >2× baseline без выхлопа в stdout — заподозрить проблему, а не ставить очередной wakeup. Проверить:
1. `ps` — процесс жив или зомби?
2. Stdout — есть ли новые строки?
3. Внешние зависимости — Docker daemon (`docker ps`), порты БД (`lsof -iTCP:5432`).
4. Сказать пользователю что аномалия, не ждать молча.

Правило: молча ждать команду больше 5 минут без выхлопа — плохо. Лучше эскалировать раньше, даже если окажется false alarm.
```
