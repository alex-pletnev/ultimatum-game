# 02. Правила игры

## Каноническая Ultimatum Game

Игра из экспериментальной поведенческой экономики (Güth et al., 1982):

1. Есть **сумма X**.
2. **Proposer** предлагает как разделить X: `(X − offer)` себе, `offer` — respondent'у.
3. **Respondent** видит offer и решает:
   - **Accept**: proposer получает `X − offer`, respondent получает `offer`.
   - **Reject**: **оба получают 0**.
4. Игра одноразовая (single-shot), 2 игрока.

Диапазон offer'а: `0 ≤ offer ≤ X` (весь X можно отдать целиком или ничего).

## Наши адаптации

Мы расширяем канон для многопользовательского формата и итеративной игры.

| Параметр | Канон | Наш проект | Комментарий |
|----------|-------|-----------|-------------|
| Игроков в сессии | 2 | N (`config.numPlayers`, 2..120) | Многопользовательская вариация |
| Роли | proposer / responder (фиксированы) | Каждый игрок в раунде — и proposer, и respondent (одновременно) | В одном раунде: игрок делает оффер И получает оффер от другого |
| Раундов | 1 (single-shot) | N (`config.numRounds`, 1..10) | Iterated variant для сбора статистики |
| Сумма | Фиксированная | `config.roundSum` (10..100000), immutable | ✓ канон |
| Правило accept | proposer keeps `X − offer`, respondent gets `offer` | То же: `(roundSum − amount, amount)` | ✓ канон |
| Правило reject | Оба получают 0 | То же: `(0, 0)` | ✓ канон |
| Диапазон offer | `0 ≤ offer ≤ X` | `0 ≤ amount ≤ roundSum` — проверяется на сервере | ✓ канон |

## Как назначаются пары (proposer → respondent)

В начале каждого раунда каждый игрок отправляет свой offer (амаунт, ещё без адресата). Когда все офферы собраны — сервер применяет **shuffle-стратегию**:

### FREE_FOR_ALL

Каждый оффер получает случайного respondent'а среди участников, с двумя ограничениями:
1. `respondent ≠ proposer` (не сам себе).
2. Каждый игрок respondent'ом ровно 1 оффера (bijection).

Для 4 игроков A, B, C, D — типичная раскладка: `A→B, B→C, C→D, D→A` (одна из возможных перестановок).

### TEAM_BATTLE

Игроки разбиты на команды (`config.numTeams`, 2..5). Ограничения:
1. `respondent ≠ proposer`.
2. `team(respondent) ≠ team(proposer)` — respondent обязательно из чужой команды.
3. Каждый игрок respondent'ом ровно 1 оффера.

## Фазы раунда

```
CREATED
   │  (админ стартовал сессию / следующий раунд)
   ▼
WAIT_OFFERS
   │  (собран последний оффер, автоматически)
   ▼
ALL_OFFERS_RECEIVED
   │  (сервер применил shuffle, автоматически)
   ▼
OFFERS_SENT              ← в это время фронт получает персональные офферы
   │  (собрано последнее решение, автоматически)
   ▼
ALL_DECISIONS_RECEIVED   ← сервер публикует счёт раунда
   │  (админ вызвал round.start / round.abort)
   ▼
FINISHED / ABORTED
```

Подробнее — [08-state-machines.md](08-state-machines.md).

## Скоринг

Считается автоматически после `ALL_DECISIONS_RECEIVED` и рассылается по WS-топику `/topic/session/{id}/scoreUpdated`.

Формула для одного оффера:
- **Accepted:** proposer получает `roundSum − amount`, respondent получает `amount`.
- **Rejected:** оба получают `0`.

Player score = сумма по всем офферам во всех раундах, где игрок был участником (proposer или respondent).

Team score (только для TEAM_BATTLE) = сумма score всех членов команды.

Payload скоринга — `SessionScoreDto` (см. [06-data-models.md](06-data-models.md)).

## Роли игроков

| Роль | Что может |
|------|-----------|
| **ADMIN** | Создать сессию, управлять фазами: start/close/open/round.start/**round.abort** |
| **PLAYER** | Присоединиться к сессии, отправлять офферы и решения |
| **OBSERVER** | Присоединиться как наблюдатель, читать все события (нельзя отправлять offer/decision) |
| ~~NPC~~ | Зарезервирована; регистрация запрещена сервером |

## Что НЕ реализовано (важно для UX)

- **Таймауты хода:** `config.timeoutMoveSec` есть в схеме, но сервер не переключает фазу автоматически по нему. Раунд «висит» пока все не отправят offer/decision, либо админ не вызовет `abortCurrentRound`. Если хочется таймер на UI — считать локально; когда истёк, либо ждать других, либо просить админа abort'нуть.
- **Pause:** нет.
- **NPC-игроки:** не реализованы.
- **Refresh-token rotation:** отключён — старый refresh продолжает работать до `exp` даже после использования.

## Терминология в API

Может отличаться от учебного канона — обращать внимание:

| В API | В доке про канон | Что имеется в виду |
|-------|------------------|--------------------|
| `Offer.offerValue` / `amount` | offer / share | Сколько предложено respondent'у |
| `Session` | game | Партия из N раундов |
| `Round` | single game | Один раунд |
| `Decision` | accept/reject | Ответ respondent'а на оффер |
| `Session.members` | players | Игроки, зарегистрированные в сессии как PLAYER |
| `Session.observers` | audience | Наблюдатели |
