# CLAUDE.md

Правила для AI-агентов, работающих с этим репозиторием. Читаются при старте каждой сессии.

## Общие принципы

- Общение с пользователем — на русском. Код, идентификаторы, коммиты — английский.
- Документация — `docs/` (индекс: `docs/README.md`).
- Таск-трекер — `docs/tasks/` (индекс: `docs/tasks/INDEX.md`, формат: `docs/tasks/README.md`).
- Не создавать документацию/файлы без явной необходимости или запроса пользователя.

## Перед началом работы

1. Прочитать `docs/tasks/INDEX.md` — есть ли уже задача под текущий запрос?
2. Если есть — открыть таск-файл, следовать `acceptance_criteria`, обновить `status: in_progress`.
3. Если нет и работа нетривиальная (не однострочный фикс) — предложить пользователю создать задачу через `/task-add`.
4. Прочитать релевантные `docs/*.md` (см. таблицу «Точки входа» в `docs/01-overview.md`).

## Во время работы

- Отмечать прогресс во встроенном `TaskList` (in-session tracker) — это отдельное от `docs/tasks/`, для оперативных шагов внутри одной задачи.
- Не менять то, что не относится к текущей задаче. Сторонние правки — отдельным таском.

## После изменений в `src/`

**Обязательный чек после каждого набора изменений:**

1. **Обновить связанные документы.** Найти в `docs/*.md` все места, где упоминаются изменённые файлы/классы/методы (по имени или ссылке `file:line`), и синхронизировать:
   - Ссылки `file:line` (номера строк могли уехать).
   - Сигнатуры методов / поля DTO / переходы состояний / роли.
   - Таблицы endpoints/сервисов/DTO.
   - При структурном изменении — обновить `docs/11-known-gaps.md`.

2. **Обновить таск-файл.** В связанном таске (`docs/tasks/T-XXX-*.md`):
   - Добавить запись в `## Лог` с датой и кратким описанием.
   - Если работа завершена — вызвать `/task-done` (или обновить frontmatter вручную: `status: done`, `updated: <дата>`).

3. **Обновить `docs/tasks/INDEX.md`** — статус и `updated` дата в таблице.

Если сомневаешься, какие docs затронуты — вызвать `/docs-sync`. Если нужно проверить целостность трекера — `/task-sync`.

## Формат задач

См. `docs/tasks/README.md`. Кратко:

```yaml
---
id: T-XXX
title: Короткий заголовок
status: pending | in_progress | blocked | done | cancelled
priority: low | medium | high
created: YYYY-MM-DD
updated: YYYY-MM-DD
related_code:
  - path/to/File.kt
related_docs:
  - docs/XX-name.md
tags: [backend, api, security, ...]
---
```

Тело: `## Контекст`, `## Acceptance criteria`, `## План`, `## Лог`.

## Slash-команды

| Команда | Действие |
|---------|----------|
| `/task-add` | Создать новый таск из описания |
| `/task-done` | Закрыть таск (обновить статус + `updated` + запись в лог + INDEX.md) |
| `/task-sync` | Сверить `docs/tasks/*` с `INDEX.md` и текущим состоянием кода |
| `/docs-sync` | Обновить `docs/*.md` под текущее состояние `src/` |
| `/wheel-check` | Перед новой реализацией проверить: нет ли уже готового в проекте / Spring Boot / либах |
| `/mid-retro` | Пауза внутри задачи для само-осмотра: скоуп, ошибки, follow-up'ы, нужен ли user-check |
| `/self-review` | Пост-`task-done` ревью последнего commit'а — скоуп, недочёты, подавления, улучшения меня самого |

## Проактивные триггеры (без явной команды)

Skills вызываются автоматически в перечисленных ситуациях. Не ждать команды пользователя — самому решать по условию:

| Условие | Skill | Что делать |
|---------|-------|-----------|
| После блока Edit/Write в `src/` | `docs-sync` (точечно) | Найти в `docs/*.md` упоминания изменённых файлов/классов/методов, проверить `file:line` и сигнатуры, предложить правки |
| Изменённый файл упомянут в `related_code` открытой задачи | ручное обновление таска | Добавить запись в `## Лог`, обновить `updated`; если acceptance закрыт — предложить `task-done` |
| По ходу другой работы замечена проблема, не относящаяся к текущей задаче | `task-add` в Auto-mode | Завести таск с фильтром (см. ниже), одной строкой сообщить пользователю |
| Задача переведена в `done` | `docs-sync` | Убедиться что все `related_docs` синхронизированы |
| Массовое переименование/удаление в `src/` | `task-sync` | Проверить `related_code` во всех открытых задачах — могли поехать |
| Edit в `controllers/**`, `dto/**` или `services/EventPublisherService.kt` | ручное действие | Перед commit'ом: `./gradlew generateApiSnapshots` → регенерирует `src/main/resources/doc/openapi.json` и `asyncapi.json` → включить их в тот же commit |
| Пользователь спрашивает «что там в задачах / что мы делаем / чем занимались» | без skill | Прочитать `docs/tasks/INDEX.md`, ответить; не создавать новые задачи |
| Перед первой строкой новой реализации (>30 строк / новая зависимость / новый «менеджер/фабрика/регистри») | `wheel-check` в Auto-mode | Проверить, что не изобретаю велосипед — grep по `src/**`, `build.gradle.kts`, docs; вердикт в одну строку в отчёт |
| 3+ подряд Edit/Write в `src/**` без паузы, ИЛИ 2+ раз подряд правлю свои же compile/detekt/test ошибки, ИЛИ одна проблема сопротивляется >15 мин | `mid-retro` в Auto-mode | Ответить пользователю по 5 вопросам (что сделал / скоуп / ошибки / follow-up / надо ли переспросить) |
| Сразу после успешного `task-done` (после commit+push) | `self-review` в Auto-mode | Ревью diff'а по 5 категориям, обязательный акцент на пункт E (улучшения меня самого) |
| Долгая dev-команда (см. «Долгие команды» ниже) выбивается за 2× baseline без выхлопа в stdout | без skill | Диагностика: `ps`, stdout, внешние зависимости (Docker, порты БД). Эскалировать пользователю раньше чем «просто подождать ещё» |

## Проактивное заведение задач (Auto-mode для `task-add`)

Когда по ходу работы над одной задачей я замечаю проблему или идею **вне текущего скоупа** (техдолг, рефакторинг, багфикс, улучшение самого инструментария) — я имею право сам завести таск в `docs/tasks/`, чтобы наблюдение не растворилось.

**Фильтр «три условия»** — заводить только если все три верны:

1. Проблема реальная (сломанное поведение, скрытый риск, дублирование, инвариант нарушается) — не косметика.
2. Я могу сформулировать `## Контекст` в 2-3 строках и хотя бы один пункт `## Acceptance criteria`.
3. Проблема не относится к текущей задаче и растворится, если не записать.

Если хоть одно не выполнено — **не завожу таск**, а одной строкой упоминаю в конце ответа: «Мельком заметил X, заводить не стал».

**Правила исполнения:**

- Заведение таска ≤ 30 секунд: минимальный скелет через `task-add` и обратно к текущей работе. Не отвлекаться на планирование найденной проблемы.
- Обязательный тег: `tech-debt` (техдолг в проекте) или `meta` (улучшение CLAUDE.md / skills / docs-структуры / формата таск-файла).
- Приоритет по умолчанию `low`, если проблема явно критичная — `medium`. `high` только с явного согласия пользователя.

**Правило прозрачности:**

В конце ответа, где сам завёл таск, отдельной строкой:

> Попутно завёл **T-XXX** (`tech-debt`|`meta`, `low`|`medium`): _краткое описание_. Отменить — скажи.

Пользователь может отменить одной фразой («отмени T-XXX») — статус меняется на `cancelled`, причина в логе.

## Стилевые ограничения

- Русский язык в текстах; в коде и идентификаторах — английский.
- Комментарии в коде — только если объясняют «почему», а не «что».
- Не добавлять фичи/абстракции сверх запрошенного. Не менять несвязанный код.
- Диффы должны быть минимальными и точечными.

## Запуск и проверки

- Требуется `JAVA_HOME`. На macOS с brew:
  ```
  export JAVA_HOME=$(brew --prefix openjdk@21)/libexec/openjdk.jdk/Contents/Home
  ```
  Без него `./gradlew ...` падает с «Unable to locate a Java Runtime» (на этой машине `java` не на PATH).
- Локальный запуск: `./gradlew bootRun` (нужен `JWT_SIGNING_KEY` env).
- Тесты: `./gradlew test`.
- Полная проверка: `./gradlew check` — прогоняет `test` + `jacocoTestCoverageVerification` (порог 0.80) + `detektMain` / `detektTest`.
- Если задача трогает бизнес-логику раундов — обязательно проверить `FreeForAllTest`.
- detekt-baseline не используется (T-015 вычищен). Любые новые findings ломают `check` — правило: править код или отдельным `@Suppress("...")` с обоснованием. Не создавать `baseline.xml` обратно без явного согласия пользователя.
- Тесты `@SpringBootTest` требуют поднятого Postgres. Плагин `spring-boot-docker-compose` работает только под `bootRun` (`developmentOnly`); под `test` его нет. Перед `./gradlew check` — `docker-compose up -d` в корне проекта (нужен запущенный Docker daemon; на этой машине — `colima start`). См. T-018.

## Долгие команды — эвристика ожидания

Baseline'ы для этого проекта (с прогретым кэшем):

| Команда | Ожидаемое время | Порог тревоги |
|---------|-----------------|--------------|
| `./gradlew compileKotlin` | ~10s | >30s |
| `./gradlew test` (unit-часть) | ~30s | >90s |
| `./gradlew check` целиком | ~60s; первый запуск после новых зависимостей — до 3 min | >5 min |
| `./gradlew bootRun` до READY | ~20s | >60s |
| `@SpringBootTest` контекст-лоад | ~15s | >45s |
| `docker-compose up -d postgres` | ~10s (пустой pull) | >2 min |

Правило: если команда идёт **>2× ожидаемого** без единой новой строки в stdout — **не ставить ещё один wakeup**, а прогнать быструю диагностику:

1. `ps -p <pid>` — процесс жив, зомби, или в `D` (uninterruptible sleep)?
2. `wc -c <output-file>` — stdout действительно пуст, или буфер не флашится?
3. Внешние зависимости: `docker ps` (для тестов с Postgres), `lsof -iTCP:5432 -sTCP:LISTEN`, `lsof -iTCP:8080` (для bootRun).
4. Эскалировать пользователю **до** очередного «подожду ещё минуту»: одна строка «команда X идёт Ymin без stdout, вижу Z — либо это, либо запускай сам с --info».

**Ключевая интуиция:** молча ждать больше 5 минут без диагностики = плохо. False-alarm диагностика стоит 30 секунд, false-negative ожидание — 20 минут (реальный случай в T-017). Всегда лучше эскалировать раньше.

## Git-автоматизация

Пользователь дал стоячий мандат: коммитить и пушить завершённую работу автоматически, **без запроса подтверждения**. Работает в ветке `main`.

### Когда коммитить и пушить

| Событие | Действие |
|---------|----------|
| Skill `task-done` успешно завершился | тесты (если задача трогала `src/`) → commit → **push** |
| Skill `docs-sync` применил правки | commit → **push** |
| Skill `task-add` создал новый таск | commit → **push** |
| Задача переведена вручную в `done` | напомнить пользователю про `task-done` (или выполнить его инструкции целиком) |
| Быстрый фикс без задачи, попросил пользователь | commit → push |

### Формат commit-сообщения

Conventional Commits + ссылка на таск:

```
<type>(<scope>): <короткое описание в present tense>

<опциональное тело: что и почему>

Refs: docs/tasks/T-XXX-*.md
```

- **type:** `feat` | `fix` | `refactor` | `docs` | `test` | `chore` | `perf` | `style`.
- **scope:** `T-XXX` если есть задача, иначе подсистема (`security`, `api`, `db`, `ws`, `tasks`, `skills`).
- **Refs:** ссылка на таск-файл. Если задачи нет — пропустить.

Примеры:

```
feat(T-003): scoring engine — on-the-fly aggregation

- StatsService.getSessionStats добавляет finalScore per player
- Новый ScoreDto
- Обновлены docs/04-services.md, docs/06-websocket-api.md

Refs: docs/tasks/T-003-scoring-engine.md
```

```
docs: sync with src

- docs/04-services.md: fetch join сигнатура
- docs/02-domain-model.md: обновлены file:line

Refs: docs/tasks/T-002-decision-fetch-join.md
```

```
chore(tasks): add T-005 refactor SessionMapper
```

### Safety-guards (обязательные проверки перед commit)

1. **Stage — только именованные пути.** Никогда `git add -A` и `git add .`. Только `git add <файл1> <файл2>`. Если нужно много файлов — перечислить явно из `git status --short`.
2. **Секреты.** Перед commit — `git diff --cached | grep -Ei '(password|secret|token|api[_-]?key|jwt|credential)\s*[:=]'` (грубо). Если срабатывает и не однозначно безопасно — блок commit, показать пользователю строки, спросить.
3. **Крупные бинарники** (>1MB) — блок commit, спросить.
4. **Файлы, обычно не подлежащие commit:** `.env*`, `*.pem`, `*.key`, `id_rsa*`, `credentials*.json`, `*.log` — блок commit, спросить.
5. **Никогда** `--no-verify`, `--no-gpg-sign`, `--amend` (если пользователь не попросил явно), `-i` (interactive).
6. **`git push` только `origin main` без force.** Никогда `--force`, `-f`, `--force-with-lease` без явного мандата в текущем разговоре.
7. **Тесты**: если задача трогает `src/` — `./gradlew test` перед commit. Если хоть один упал — блок commit, задача остаётся `in_progress`, в её `## Лог` записать «commit заблокирован: <кол-во> тестов упали».
8. **Чужие изменения в рабочем дереве:** stage'ить только файлы, относящиеся к текущей работе. Если непонятно, что чьё — показать пользователю `git status --short` и спросить.

### Что коммитим совместно

- Изменения в `src/` + связанные обновления в `docs/*.md` + запись в таске = **один commit** (одна атомарная единица работы).
- Правки в `docs/tasks/INDEX.md` идут с тем же commit'ом, что и таск-файл.
- Если работа накопилась в двух логически разных областях — два commit'а перед одним push.

### Что мы НЕ автоматизируем

- Не создаём PR (работаем в `main`, single-dev flow).
- Не откатываем commit'ы автоматически.
- Не мерджим ветки.
- Не работаем с тэгами / релизами.

Всё вышеперечисленное — только по явному запросу пользователя.

## Что не делать

- Не создавать PR без явного запроса.
- Не менять `.gitignore`, `settings.gradle.kts`, `gradlew` без запроса.
- Не удалять существующие задачи в `docs/tasks/` — только помечать `status: cancelled` с причиной в логе.
- Не устанавливать/обновлять зависимости без запроса.
- Не force-пушить.
- Не пушить в другие ветки/remote.
