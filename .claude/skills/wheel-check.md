# wheel-check

Прогон 3–5-минутной проверки перед новой реализацией: «а не изобретаю ли я велосипед».

Смысл: чаще всего то, что кажется «надо написать самому», уже есть — либо в проекте, либо в фреймворке, либо в одной из уже подключённых зависимостей. Стоимость проверки — минуты, стоимость велосипеда — часы + вечный техдолг.

## Аргументы

- **Explicit:** `/wheel-check <короткое описание того, что собираюсь реализовать>`.
- **Auto-mode:** без аргументов, скилл сам знает контекст текущего шага.

## Шаги

1. **Сформулировать в одну фразу**, что именно я собираюсь реализовать. Написать пользователю (не более 15 слов).

2. **Проверить проект:**
   - `Grep` по `src/main/kotlin/**` на 2–3 ключевых слова из формулировки. Пример: собираюсь «request-id filter» → грепаю `RequestId|CorrelationId|MDC.put`.
   - Прочитать `docs/04-services.md` — есть ли уже сервис с похожей ролью.
   - Прочитать `docs/07-dto-and-mappers.md` (для DTO/маппинга) или `docs/08-security.md` (для auth/фильтров).

3. **Проверить Spring Boot / фреймворки:**
   - Смотрю `build.gradle.kts` — какие starter'ы и либы уже подключены.
   - Для типовых нужд — соотнести с известными механизмами: `micrometer-tracing` (traceId в MDC), `spring-boot-actuator` (health/metrics), `spring-security-messaging` (WS auth), `MapStruct` (DTO-маппинг), `springdoc` (OpenAPI), `springwolf` (AsyncAPI). Если моя задача накрывается одним из них — велосипед не строим.

4. **Проверить идиомы Kotlin/JVM:**
   - Для thread-local контекста — `MDC` (SLF4J), `ThreadLocal`, `kotlin.coroutines` context.
   - Для scope-based cleanup — `.use { }` / `AutoCloseable` / `try-finally`.
   - Для короткоживущих объектов — `data class`, `sealed interface`.
   - Не писать «менеджер» / «регистри» / «фабрику» без сильной причины — это часто и есть велосипед.

5. **Вывести вердикт** — 3 варианта в отчёте пользователю:
   - **«Пиши сам».** Обосновать: почему готовое не подходит, что именно уникально.
   - **«Используй X».** Указать конкретную либу/класс/паттерн, коротко чем заменяет мою идею.
   - **«Гибрид».** X делает Y%, остаток — своими руками. Уточнить что именно.

6. **Отчёт пользователю** — 3–7 строк. Один из трёх вердиктов + короткое обоснование. Не расписывать шаги «как я искал» — сразу к выводу.

7. **Только после отчёта — писать код.** Если вердикт «используй X» — соответственно править план.

## Auto-mode: агент запускает wheel-check сам

Скилл вызывается **до** написания кода, если срабатывает хотя бы одно из условий:

- Собираюсь добавить новую зависимость в `build.gradle.kts` (любую).
- Собираюсь создать новый файл в `src/main/kotlin/**` длиннее чем ~30 строк.
- Собираюсь написать «менеджер», «регистри», «фабрику», «диспетчер», «пул», «кэш», «валидатор», «конвертер» — любое существительное с суффиксом «-er/-or» / «-Registry» / «-Factory» / «-Manager». Это классический маркер велосипеда.
- Собираюсь реализовать что-то, что типично покрывается фреймворком (MDC/tracing, health-check, метрики, sec-фильтр, кеширование, JSON-сериализация, валидация, i18n, миграции).

Если срабатывает — молча провести проверку (шаги 1–5) и в отчёте пользователю указать одной строкой: «Прогнал wheel-check: [вердикт]». Дальше действовать по вердикту.

## Что НЕ является поводом для wheel-check

- Однострочные фиксы, правки логов, добавление поля в DTO, изменение сообщения ошибки.
- Правки тестов.
- Правки `docs/*.md`.
- Rename переменной / файла.
- Отдельный маленький util-хелпер (<10 строк).

## Ограничения

- Проверка занимает максимум 3–5 минут. Если ушёл дольше — либо задача сложнее и стоит проговорить с пользователем, либо я слишком глубоко копаю: закрыть проверку с ближайшим найденным вариантом.
- Не переоценивать чужие либы. Если готовое требует 5+ конфигов и 2+ адаптера — «пиши сам» может быть правильным ответом. Указать это в отчёте.
- Не подменять `wheel-check` полноценным дизайном. Это gate «не велосипед», а не «правильная архитектура».
